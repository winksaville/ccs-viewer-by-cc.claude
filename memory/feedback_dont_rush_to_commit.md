---
name: Don't rush to commit in new areas
description: Work back and forth with user before committing, especially for new features — don't jump straight to commit
type: feedback
---

Don't jump straight to committing after implementing something,
especially in a new area. Work back and forth with the user to
iterate on the result first. Only commit when the user asks or
when I ask and get explicit approval.

**Why:** New features need iteration. The user expects a
collaborative review loop before finalizing, not an immediate
jump to the commit workflow.

**How to apply:** After implementing, show the result and discuss.
Wait for the user to say "commit" or ask "ready to commit?" Only
then start the commit workflow.
